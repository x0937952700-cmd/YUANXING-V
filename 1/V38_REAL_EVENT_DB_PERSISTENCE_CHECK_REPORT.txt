V38 真實對照與修正報告

一、這次確認到的事件問題
1. 庫存 / 訂單 / 總單「確認送出」不是單純按鈕沒反應：
   - module.html 的 #submit-btn 原本會呼叫 confirmSubmit(event)。
   - 各頁 v22 JS 裡也有自己的 submit handler。
   - V34 / V35 / V36 又追加 submit lock，舊新版 handler 疊在一起時，畫面可能有點擊但中途被 stopPropagation / 錯誤 catch / 後端錯誤中斷。
2. V37 仍載入 yx_v36_submit_backend_fix.js，版本命名與最後接管不一致，導致後續排查容易以為 V37 已接管，但實際最後檔案仍是 V36。
3. 後端 /api/ship 在 ship_order 成功後，build_customer_payload_snapshot 如果遇到舊欄位或快照錯誤，仍可能把已完成的出貨回覆成失敗。

五、仍需你部署後檢查的重點
1. Render 環境是否真的連到 PostgreSQL DATABASE_URL。
2. 部署後請先硬重新整理或清 PWA 快取，因為舊版曾用過 service worker。
3. 若 PostgreSQL 既有資料表有非常舊的欄位結構，V38 會 fallback 寫入現有欄位，但建議打開 /api/health?force=1 確認 migration 已補完。
